## 2026-07-11

### 错误 1：waitKey 理解错误（代码能跑，但理解全错）

- 出处：打开摄像头.py `if key & 0xFF == ord('q')`
- 类型：概念理解错误
- 当时错的理解：以为 cv2.waitKey 是"读取图像 / 按性能调快慢"，ord('q') 是"对应硬件端口"。
- 正确答案：
  - `waitKey(n)` = 等待键盘按键 n 毫秒；返回所按键的整数编码，没按返回 -1。它不碰图像。
  - 读图像是 `cap.read()` 干的（返回 frame）。waitKey 和读图像是两个函数、两回事。
  - `ord('q')` = 字符 'q' 的编码，即数字 113。
  - `& 0xFF` = 取低 8 位，把返回值洗干净，保证和 ASCII 码正确比较。
- 正确的理解方式：
  - 读函数名字面意思：waitKey = wait key = 等待按键。
  - 做实验看事实：打印返回值，不按 = -1，按 q = 113，与 ord('q') 一致，真相立现。
- 如何降低复发概率：
  1. 不懂就 print 出来看，眼见的事实碾压脑补的猜测。
  2. 先读函数 / 变量名的字面意思。
  3. 用自己的话讲一遍；冒出"应该是 / 为了…吧"就警觉，那是在猜不是懂。

### 错误 2：缩进不一致（IndentationError）

- 出处：打开摄像头.py 循环体内新加的两行
- 类型：语法错误
- 当时错的做法：新加行缩进 8 空格，循环体其他行是 3 空格。
- 正确答案：同一代码块每行缩进必须完全一致（推荐统一 4 空格）。
- 正确的理解方式：Python 没有大括号，完全靠缩进判断代码归属，缩进乱 = 解释器不知道这行归谁。
- 如何降低复发概率：加新行时与上一行左对齐；见 IndentationError 先查"哪行缩进和邻居不一样"；编辑器统一缩进设置。

### 错误 3：改代码没改全，waitKey 被调用两次

- 出处：打开摄像头.py，加了 `key = cv2.waitKey(1)` 却没删旧的 `if cv2.waitKey(1)...`
- 类型：修改不彻底，新旧代码并存
- 当时错的做法：只顾加新行，忘了改旧行，一圈循环调了两次 waitKey。
- 正确答案：返回值存进 key 一次，之后都用 key；旧的重复调用删掉。
- 正确的理解方式：有副作用的函数（waitKey 每次调用都消耗一次事件）一圈只调该调的次数，取一次存起来复用。
- 如何降低复发概率：改完通读整段，确认新旧写法没有同时存在；心里数一遍"这函数一圈该调几次"。

## 2026-07-12

### 错误 4：用了老版 MediaPipe API（mp.solutions 不存在）

- 出处：骨架显示.py 初稿
- 类型：API 版本混淆
- 当时错的做法：写了 `mp.solutions.pose.Pose()` 这种旧版 API。
- 正确答案：MediaPipe 0.10.35 已删除 `mp.solutions`，改用 task-based API：
  - `from mediapipe.tasks.python.vision import PoseLandmarker, PoseLandmarkerOptions`
  - `options = PoseLandmarkerOptions(base_options=..., running_mode=...)`
  - `landmarker = PoseLandmarker.create_from_options(options)`
  - 送帧：`landmarker.detect_for_video(mp_image, timestamp_ms)`
- 正确的理解方式：看到新版本号（0.10.x）第一反应是查文档看 API 有没有变，不要假定和之前用过的版本一样。
- 如何降低复发概率：装完库先 `pip show mediapipe` 看版本号；导入前先用 `dir()` 查模块里有哪些可用的类。

### 错误 5：中文路径传 C 库导致 FileNotFoundError

- 出处：骨架显示.py 初始化 PoseLandmarker 时报错
- 类型：编码不匹配（Python UTF-8 vs C 库 GBK）
- 当时错的做法：写绝对路径 `D:/Study/项目/fall-detection/仓库/pose_landmarker_lite.task`，里面含中文。
- 正确答案：改用纯 ASCII 相对路径 `pose_landmarker_lite.task`，脚本开头加 `os.chdir(os.path.dirname(os.path.abspath(__file__)))` 确保相对路径从脚本目录找。
- 正确的理解方式：Python 传字符串给 C 动态库时，编码可能不兼容。底层用 C 写的库（MediaPipe、OpenCV 等）在 Windows 中文系统上尤其容易踩这个坑。根本原因是 Python 3 用 UTF-8，Windows C runtime 默认用系统 ANSI (GBK)。
- 如何降低复发概率：项目路径尽量全英文字母，新建项目时优先起英文名；需要传路径给底层库时优先用相对路径。

### 错误 6：VSCode 解释器没选对，找不到 venv 里的库

- 出处：VSCode 按运行按钮跑脚本，报 `ModuleNotFoundError: No module named 'mediapipe'`
- 类型：环境配置错误
- 当时错的做法：VSCode 右下角没切换解释器，默认用了系统 Python（C:\Python314\），但 mediapipe 装在项目 venv 里。
- 正确答案：`Ctrl+Shift+P` → `Python: Select Interpreter` → `Enter interpreter path...` → 手动浏览到 `.venv/Scripts/python.exe`。
- 正确的理解方式：venv 的隔离 = 每个环境有自己的 site-packages。VSCode 的运行按钮用哪个 Python，就去哪个 Python 的 site-packages 找库。解释器不指向 venv，就找不到 venv 里装的库。VSCode 自动扫描覆盖不全层级深/含中文的 venv 路径，需要手动指。
- 如何降低复发概率：新项目第一件事：设好解释器再写代码；右下角确认 Python 版本旁有 `('.venv': venv)`。

### 错误 7：把终端和 Python REPL 搞混了

- 出处：在 PowerShell 输入 `python.exe` 回车，进入 `>>>` REPL 后尝试运行脚本路径
- 类型：工具使用概念混淆
- 当时错的做法：分两步——先启动 Python 交互环境，再在 `>>>` 里输入脚本路径。
- 正确答案：一整行命令 `python.exe 骨架显示.py`，不要中途回车。`>>>` 是 Python 的交互式解释器（一行一行执行 Python 代码），不是终端（运行命令/程序）。脚本路径不是合法 Python 语句，所以报 SyntaxError。
- 如何降低复发概率：看命令提示符判断自己在哪——`PS >` = 终端，`>>>` = Python REPL。出现 `>>>` 就输 `exit()` 退出，回到终端再输完整命令。
